# Performance Test Cases — NexaPay Payment Gateway

**Platform under test:** the (fictional) NexaPay payment gateway API and its downstream ledger/webhook pipeline.

These test cases outline the performance-testing approach a QA/performance engineer would apply to a payment gateway, using representative tooling such as **k6**, **JMeter**, or **Gatling** to drive load, and a metrics stack (e.g., Grafana/Prometheus-style dashboards) to observe latency, error rate, and saturation. All numeric targets below (TPS, latency, durations) are **illustrative example SLA targets used for this exercise**, not claimed production metrics of any real system — they exist to show the *kind* of target and pass/fail criteria a performance test plan would define, not to represent measured results.

**Total test cases:** 21

| PERF-001 | Baseline authorize throughput | Steady load ramping to an example target of 200 transactions/sec (TPS) over 10 minutes, sustained for 15 minutes. | Mixed card BINs, single currency (AED), 95% approvals / 5% simulated declines. | System sustains the target TPS with error rate ≤ 0.5% and no unbounded queue growth; example SLA: authorize p95 latency ≤ 400ms at target load. | P1 |
| PERF-002 | Authorize endpoint latency SLA (p95/p99) | Constant load at ~70% of baseline target TPS for 20 minutes. | Standard authorize payloads, valid tokens/cards. | Example SLA targets: p95 ≤ 400ms, p99 ≤ 800ms end-to-end (excluding client network); latencies remain stable (no upward drift) across the full window. | P1 |
| PERF-003 | Sustained load / soak test | Constant load at ~60% of baseline target TPS for an extended window (e.g., 4-8 hours). | Continuous mixed authorize/capture/refund traffic. | No memory growth, connection-pool exhaustion, or gradual latency degradation over the soak window; error rate remains flat and low throughout. | P1 |
| PERF-004 | Spike test — flash-sale burst | Traffic spikes from baseline to 5x baseline TPS within 30 seconds, held for 5 minutes, then returns to baseline. | Authorize requests concentrated on a small set of popular merchant SKUs/orders. | System auto-scales or gracefully sheds/queues excess load without crashing; error rate during the spike stays within an agreed elevated-but-bounded threshold (e.g., ≤ 2%); recovery to normal latency within 2 minutes of spike end. | P1 |
| PERF-005 | Degraded downstream card-network/processor response time | Inject artificial latency (e.g., 3-5 seconds) into the simulated downstream processor for a subset of requests. | Authorize requests routed through the degraded processor path. | Gateway applies its own timeout (e.g., 8s) and returns a clear timeout/decline response rather than hanging indefinitely; no thread/connection pool exhaustion cascades to unrelated requests. | P1 |
| PERF-006 | Downstream processor total outage / fallback behavior | Simulated processor is made completely unreachable. | Authorize requests during the outage window. | Requests fail fast with a clear `PROCESSOR_UNAVAILABLE` error (not a hang or generic 500); if a fallback/secondary processor path exists, it is exercised; system recovers automatically once the processor is reachable again. | P2 |
| PERF-007 | Concurrent refund requests on the same order (race condition check) | Two to five concurrent refund requests submitted against the same captured transaction within milliseconds of each other. | Single transaction, amount 5000, all requests requesting full refund. | Exactly one refund is processed; remaining concurrent requests are rejected with a conflict/already-refunded error; no double-refund occurs under load. | P1 |
| PERF-008 | Concurrent capture requests on the same authorization | Multiple concurrent capture requests against the same AUTHORIZED transaction. | Single transaction, amount 5000. | Exactly one capture succeeds; others receive a conflict response; ledger shows a single capture entry, not multiple. | P1 |
| PERF-009 | Webhook queue backpressure under peak notification volume | A burst of status-change events (e.g., 10x normal volume) is generated within a short window, each requiring a webhook delivery attempt. | Simulated batch of captures/refunds completing simultaneously. | Webhook delivery queue absorbs the burst without dropping events; delivery lag increases temporarily but drains back to baseline within an agreed window (e.g., 10 minutes); no events are silently lost. | P2 |
| PERF-010 | Database write throughput under peak transaction volume | Sustained peak load profile hitting the transactions/ledger tables directly (bypassing API-level throttling for this test). | High-frequency inserts/updates to `transactions` and `ledger_entries`. | Write latency remains within an acceptable bound; no lock contention causing timeouts; replication lag (if applicable) stays within an agreed threshold. | P2 |
| PERF-011 | Recurring/tokenized charge batch performance | A scheduled batch job charges a large set of saved tokens (e.g., 50,000 subscriptions) within a defined nightly window. | Bulk token-charge job against a test dataset of saved payment methods. | Batch completes within the defined processing window; per-charge failures are isolated and retried/reported individually without halting the entire batch. | P2 |
| PERF-012 | Rate-limit enforcement under high request volume from a single API key | A single merchant API key sends requests well above its configured rate limit. | Burst of requests exceeding the documented per-key rate limit. | Excess requests receive `429 Too Many Requests` with a `Retry-After` header; legitimate traffic from other merchants/keys is unaffected (no noisy-neighbor impact). | P2 |
| PERF-013 | Graceful degradation under overall platform overload | Load is increased well beyond the system's provisioned capacity (stress test to find the breaking point). | Load ramped until error rate or latency SLA is clearly breached. | System degrades predictably (e.g., sheds load via 503/429 with backpressure) rather than failing catastrophically or corrupting data; recovery is clean once load subsides. | P3 |

## NexaPay Wallet — Performance

These test cases extend the same performance-testing approach to **NexaPay Wallet**, the consumer digital-wallet product (top-up, P2P transfer, bill payment). As above, all numeric targets are illustrative example SLAs, not claimed production metrics.

| TC ID | Scenario | Load Profile | Test Data | Expected Result / SLA | Priority |
|---|---|---|---|---|---|
| PERF-WAL-001 | P2P transfer throughput under sustained load | Steady load ramping to an example target of 150 transfers/sec over 10 minutes, sustained for 15 minutes. | Mixed sender/recipient wallet pairs, single currency (AED), 98% success / 2% simulated insufficient-balance rejections. | System sustains the target TPS with error rate ≤ 0.5% (excluding intentional rejections) and no unbounded queue growth; example SLA: transfer p95 latency ≤ 500ms at target load. | P1 |
| PERF-WAL-002 | P2P transfer latency SLA (p95/p99) under load | Constant load at ~70% of baseline target TPS for 20 minutes. | Standard P2P transfer payloads between active wallets. | Example SLA targets: p95 ≤ 500ms, p99 ≤ 900ms end-to-end; latencies remain stable (no upward drift) across the full window. | P1 |
| PERF-WAL-003 | Concurrent-transfer race safety: two simultaneous transfers from the same wallet do not double-spend | Two concurrent full-balance P2P transfer requests fired within milliseconds against the same wallet. | Single wallet, balance 5000, two concurrent transfer requests of 5000 each. | Exactly one transfer succeeds; the other is rejected with an insufficient-balance/conflict error; wallet balance never goes negative under load. | P1 |
| PERF-WAL-004 | Wallet top-up throughput under peak load | Traffic ramps from baseline to 3x baseline top-up volume within 1 minute, held for 5 minutes. | Mixed card and bank-transfer top-up requests. | System absorbs the peak without crashing; error rate stays within an agreed elevated-but-bounded threshold (e.g., ≤ 2%); recovery to normal latency within 2 minutes of peak end. | P2 |
| PERF-WAL-005 | Bill payment queue backpressure at peak volume | A burst of bill-payment submissions (e.g., 10x normal volume) is generated within a short window, e.g., around a common billing due date. | Simulated batch of bill payments across multiple billers completing near-simultaneously. | Bill-payment processing queue absorbs the burst without dropping requests; processing lag increases temporarily but drains back to baseline within an agreed window (e.g., 15 minutes); no payment is silently lost or double-processed. | P1 |
| PERF-WAL-006 | Bill payment degraded biller/processor response time | Inject artificial latency (e.g., 3-5 seconds) into the simulated downstream biller-confirmation service for a subset of requests. | Bill payment requests routed through the degraded biller path. | Wallet service applies its own timeout and returns a clear pending/timeout status rather than hanging indefinitely; no thread/connection pool exhaustion cascades to unrelated wallet requests. | P2 |
| PERF-WAL-007 | Rate-limit enforcement on the P2P transfer endpoint under burst load | A single wallet account sends P2P transfer requests well above its configured rate limit. | Burst of transfer requests exceeding the documented per-wallet rate limit. | Excess requests receive `429 Too Many Requests` with a `Retry-After` header; legitimate traffic from other wallets is unaffected (no noisy-neighbor impact). | P2 |
| PERF-WAL-008 | Wallet balance read consistency under high concurrent read/write load | Wallet balance is read repeatedly while concurrent top-up and transfer writes are in flight against the same wallet. | High-frequency `GET /v1/wallet/balance` calls concurrent with top-up/transfer writes. | Balance reads never reflect a partially-committed write (no dirty reads); each read reflects a fully committed, consistent state; no stale-balance window exceeds an agreed threshold. | P2 |

## Notes

This suite demonstrates the categories, load profiles, and pass/fail framing a QA engineer would define for a payment platform's performance test plan. It is written for a fictional platform (NexaPay) and does not represent measured results, capacity figures, or SLAs of any real production system.
