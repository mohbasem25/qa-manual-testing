# BUG-001: Rapid double-click on "Apply" promo button applies discount twice

| Field | Value |
|---|---|
| **Severity** | High |
| **Priority** | P1 |
| **Environment** | Chrome 126, Windows 11; QA/Staging (`checkout-staging.shopsphere.example`); Build v2.4-rc3 |
| **Found via** | TC-PROMO-007 |
| **Status** | Open |

## Preconditions

- Shopper is logged in with a cart subtotal of $100.00.
- Promo code `SAVE10NOW` is active, unused by this account, and configured for 10% off merchandise subtotal.

## Steps to Reproduce

1. Navigate to Order Review (or Cart) and enter `SAVE10NOW` in the promo code field.
2. Double-click the "Apply" button as quickly as possible (or click twice within ~300ms — reproducible reliably on a throttled "Fast 3G" network profile in dev tools, which widens the race window).
3. Observe the discount line and order total once the UI settles.

## Expected Result

The discount is applied exactly once: `-$10.00`, regardless of how many times "Apply" is clicked while the first request is in flight. The button should be disabled or debounced after the first click until the response returns.

## Actual Result

Two separate "apply promo" requests are sent to the backend before the button is disabled. Both are processed as valid, independent applications of the same code. The discount line shows `-$10.00` visually (single line item), but the order total is reduced by `$20.00` total — confirmed by checking the itemized total in Order Review against a manual recalculation. Re-applying the same single-use code a second time should have been blocked by the "already used" rule but was not, because both requests were issued and accepted within the same short window before the "used" flag was persisted.

## Root Cause / Notes (QA Hypothesis)

Likely a classic **check-then-act race condition**: the promo "already used" check (read) and the discount-application/usage-flag write are not wrapped in an atomic transaction or protected by a unique constraint (e.g., a `(account_id, promo_code)` unique index or a distributed lock). Two near-simultaneous requests both pass the "not yet used" check before either has written the "used" record. Recommend:
1. Adding a database-level unique constraint on promo redemption per account/order.
2. Debouncing/disabling the "Apply" button immediately on click (client-side mitigation only — must be paired with #1, since client-side alone does not protect against non-browser clients or scripted double-submits).
