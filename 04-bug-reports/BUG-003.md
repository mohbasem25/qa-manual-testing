# BUG-003: Order total mismatch between Order Review and Confirmation for non-USD currencies

| Field | Value |
|---|---|
| **Severity** | Critical |
| **Priority** | P1 |
| **Environment** | Safari 17, macOS 14; QA/Staging; Build v2.4-rc3; Account locale set to `en-GB` / currency `GBP` |
| **Found via** | TC-REV-003, TC-CONF-005 |
| **Status** | Open |

## Preconditions

- Shopper account locale/currency preference is set to GBP (non-default currency; platform default is USD).
- Cart subtotal in GBP display: £71.00 (converted from a USD-priced catalog at the session's cached FX rate).

## Steps to Reproduce

1. Add items to cart and proceed through checkout with account currency set to GBP.
2. Apply shipping (Standard, £4.99) — no promo code.
3. Reach Order Review and note the displayed grand total and currency symbol.
4. Click "Place Order."
5. On the Confirmation page, note the displayed grand total and currency symbol.
6. Compare the two values.

## Expected Result

The Confirmation page total and currency symbol should be **identical** to what was shown and agreed to at Order Review (£78.10 in this test run), since the charge should be locked in at the moment of order submission.

## Actual Result

- Order Review displayed: **£78.10**
- Confirmation page displayed: **£79.42**

The two values differ by £1.32. Investigation via the network tab shows Order Review computes the total client-side using a cached FX rate fetched at the start of the session, while the Confirmation page re-fetches (or the backend re-computes) the total using a live FX rate at the moment the order is finalized — and the underlying charge in the payment gateway sandbox matches the **Confirmation** amount (£79.42), not the amount the customer agreed to on Order Review. This means the customer is shown one price and charged a different one.

## Root Cause / Notes (QA Hypothesis)

Strong suspicion of an **FX-rate staleness/consistency bug**: the price shown at Order Review is not "locked" (e.g., via a short-lived quote/rate-lock token tied to the order) before submission. Instead, the backend appears to re-resolve the currency conversion at order-creation time using whatever FX rate is current then, which can drift from the rate used to render Order Review if even a few minutes pass or if a rate refresh job runs in between.

This is a **billing-accuracy and regulatory-risk issue** (charging a different amount than was disclosed to the customer), not merely cosmetic, hence Critical severity. Recommend:
1. Generating a rate-locked quote token when Order Review is rendered, valid for a short window (e.g., 15 minutes), and using that exact locked amount at order submission regardless of live FX movement.
2. If the lock expires, re-prompting the customer with the new total for explicit re-confirmation rather than silently charging a different amount.
3. Auditing all non-USD locales for the same discrepancy, and auditing historical staging orders for evidence of the same drift.
