# BUG-005: CVV field accepts non-numeric characters on Firefox

| Field | Value |
|---|---|
| **Severity** | Low |
| **Priority** | P3 |
| **Environment** | Firefox 128, Windows 11; QA/Staging; Build v2.4-rc3 (not reproducible on Chrome 126 or Safari 17 — Firefox-specific) |
| **Found via** | TC-PAY-006 |
| **Status** | Open |

## Preconditions

- Shopper is on the Payment step with the Credit/Debit Card form active.

## Steps to Reproduce

1. Click into the CVV field.
2. Type `12a` directly via keyboard.
3. Separately, paste the string `1 2b` (with a space and a letter) using Ctrl+V from clipboard.
4. Click "Place Order" (with an otherwise valid card number and expiry).

## Expected Result

The CVV field should accept digits only. Typed non-numeric keystrokes should be blocked in real time, and pasted content should be sanitized to strip non-digit characters, leaving only `12` in both cases. If a non-numeric value somehow reaches submission, client-side validation should block the request with an inline error.

## Actual Result

- Direct typing of `12a`: the `a` **is blocked** correctly (typed-keystroke filtering works as expected).
- Pasting `1 2b` via Ctrl+V: the full string `1 2b` **is accepted verbatim** into the field, including the space and the letter. The field's `maxlength` attribute does not trim it because paste events bypass the same keystroke-filtering logic used for typed input.
- Clicking "Place Order" with `1 2b` in the CVV field sends the literal string to the gateway sandbox, which responds with a generic decline ("invalid CVV format") rather than the UI catching it beforehand. The user sees a decline message instead of a clear, immediate inline validation error, which is a minor but avoidable UX/data-quality gap.

## Root Cause / Notes (QA Hypothesis)

The CVV field's input sanitization is implemented via a `keypress`/`keydown` event filter, which does not fire for paste (`paste` event) or drag-drop text insertion. This is a common gap when input restriction is done only via keystroke interception rather than a unified `input` event handler with regex sanitization applied to the field's resulting value regardless of input method. Severity is Low because: (a) it does not bypass server/gateway-side validation — the transaction is still correctly declined, just with a less helpful message; (b) it requires a deliberate paste action, not typical user behavior. Recommend:
1. Sanitizing on the `input` event (post-paste value) rather than only on `keydown`, stripping any non-digit character immediately.
2. If a non-numeric value is ever detected pre-submission, show the existing inline "CVV must be numeric" message rather than relying on the gateway's generic decline.

## Status

**Open** — low priority; scheduled for the next minor UI-polish pass, not a release blocker.
