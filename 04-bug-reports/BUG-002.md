# BUG-002: Postal code field accepts invalid format for US addresses

| Field | Value |
|---|---|
| **Severity** | Medium |
| **Priority** | P2 |
| **Environment** | Firefox 128, macOS 14; QA/Staging; Build v2.4-rc3 |
| **Found via** | TC-ADDR-003 |
| **Status** | Open |

## Preconditions

- Shopper is on the "Add new address" form with Country set to "United States."

## Steps to Reproduce

1. Fill in all address fields with valid data except ZIP.
2. Enter `9410` (4 digits) into the ZIP/Postal Code field.
3. Click "Save & Continue."
4. Repeat with `ABCDE` (5 letters) in a separate attempt.
5. Repeat with `94105-` (5 digits plus a trailing hyphen, no suffix digits).

## Expected Result

The form should reject all three inputs client-side (before any network call) with an inline error such as "Enter a valid ZIP code," matching either the 5-digit (`NNNNN`) or ZIP+4 (`NNNNN-NNNN`) pattern.

## Actual Result

- `9410` (4 digits): **Rejected correctly.**
- `ABCDE`: **Incorrectly accepted.** The form submits successfully and the address is saved with `ABCDE` as the stored postal code.
- `94105-` (trailing hyphen, no suffix): **Incorrectly accepted.** Form submits; address saved with the malformed value.

Downstream, the Shipping Method step still loads (shipping cost appears to fall back to a default/flat rate rather than a ZIP-based calculation), suggesting the invalid postal code silently breaks a rate lookup rather than surfacing an error to the user.

## Root Cause / Notes (QA Hypothesis)

The client-side validation regex likely only checks for a minimum/maximum character count (`{4,10}`) rather than a proper pattern anchor for digits (`^\d{5}(-\d{4})?$`). This would explain why purely numeric-but-wrong-length input (`9410`) is caught (fails length) while alphabetic input of an "acceptable" length (`ABCDE`, 5 chars) slips through. Recommend:
1. Fixing the client-side regex to `^\d{5}(-\d{4})?$` for US addresses (and equivalent country-specific patterns for other locales).
2. Adding a matching server-side validation as defense-in-depth — client-only validation is bypassable.
3. Confirming what happens downstream (shipping/tax calculation) when a malformed ZIP is already persisted for existing accounts, as a data-cleanup concern.
