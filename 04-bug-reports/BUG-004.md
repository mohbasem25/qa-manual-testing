# BUG-004: Browser back-button after confirmation can re-trigger order submission

| Field | Value |
|---|---|
| **Severity** | Critical |
| **Priority** | P1 |
| **Environment** | Chrome 126, Windows 11; QA/Staging; Build v2.4-rc3 |
| **Found via** | TC-CONF-003; Exploratory Charter 1 ("Checkout State Resilience Under Navigation") |
| **Status** | Open |

## Preconditions

- Shopper has just completed a successful order and is viewing the Confirmation page (order `SS-20260703-004821`).

## Steps to Reproduce

1. On the Confirmation page, click the browser's "Back" button.
2. The browser renders the Order Review page **from the bfcache (back-forward cache)** rather than re-fetching it — the "Place Order" button is visible and appears enabled, showing the same order details as before submission.
3. Click "Place Order" again.
4. Observe network activity and the resulting page/order state.

## Expected Result

Either (a) the Back navigation should detect the already-completed order and redirect forward to the existing Confirmation page (order state is final and should not be re-enterable), or (b) if Order Review is somehow shown, clicking "Place Order" again must be rejected server-side (e.g., via an idempotency key tied to the original checkout session) without creating a new order or new charge.

## Actual Result

Clicking "Place Order" from the bfcache-restored Order Review page successfully submits a **second, brand-new order** with the same cart contents, and the sandbox gateway processes a **second charge**. A new order number (`SS-20260703-004822`) is generated. This was reproduced 3 out of 3 attempts on Chrome and 2 out of 3 attempts on Firefox (Firefox occasionally forced a full reload instead of serving from cache, which happened to avoid the bug incidentally — not a reliable mitigation).

## Root Cause / Notes (QA Hypothesis)

Two compounding issues:
1. **Client-side:** the page does not invalidate itself from the bfcache after a successful submission (e.g., via a `pageshow` event handler checking `event.persisted` and forcing a re-fetch, or `Cache-Control: no-store` on the Order Review route).
2. **Server-side (the more serious gap):** there is no idempotency key associated with the checkout submission. Each "Place Order" click appears to be treated as an independent, fully valid new order request as long as the cart session is still valid — there's no check for "has this exact cart/session already been successfully charged in the last N minutes."

This is Critical because it can directly cause a customer to be charged twice for one intended purchase — a severe trust and financial-harm issue. Recommend:
1. Generating a unique idempotency key per checkout attempt (created once, when Order Review first loads) and requiring the backend to reject/no-op a second "Place Order" call carrying the same key.
2. Adding `Cache-Control: no-store` (or equivalent SPA route guard) to the Order Review and Payment routes.
3. Also verifying this scenario against direct URL re-entry and multi-tab duplication (related exploratory notes).

## Status

**Open** — escalated as a release blocker; highest-priority item in the exit report.
