# BUG-006: "Place Order" button remains clickable during processing, allowing double-submit

| Field | Value |
|---|---|
| **Severity** | High |
| **Priority** | P1 |
| **Environment** | Chrome 126, Windows 11; QA/Staging; Build v2.4-rc3; also reproduced on a throttled "Slow 3G" network profile (widens the reproduction window) |
| **Found via** | TC-REV-004; Exploratory Charter 2 ("Payment Form Edge Behavior & Error Recovery") |
| **Status** | In Progress (fix under review) |

## Preconditions

- Shopper is on Order Review with a valid, complete order ready to submit.
- Network conditions are simulated as "Slow 3G" in browser dev tools to widen the request round-trip window (the bug is present on normal networks too, but harder to hit manually without automation).

## Steps to Reproduce

1. On Order Review, click "Place Order" once.
2. Within roughly 1–2 seconds (before the network response returns), click "Place Order" again at the same location (the button does not visibly disable or show a spinner immediately).
3. Wait for both requests to resolve and check "My Orders" / the payment gateway sandbox transaction log.

## Expected Result

The first click should immediately disable the button (and/or replace its label with a loading indicator), making the second click a no-op with no additional network request sent. Exactly one order and one charge should result from the interaction.

## Actual Result

The button's disabled state is only applied inside the `.then()` callback of the submit request's promise chain, which does not execute until *after* the request is dispatched — there is a window of roughly 150–400ms (longer under throttled network conditions) during which the button is fully clickable. In this window, a second click dispatches a second, independent "create order" request. Both requests were observed to succeed independently in the network tab, producing two order numbers and two separate charges in the gateway sandbox for the same cart. Reproduced 4 out of 5 attempts under throttled network; 1 out of 10 attempts on a normal/fast connection (still non-zero, so not safe to deprioritize).

## Root Cause / Notes (QA Hypothesis)

The disable-button logic is sequenced incorrectly: it should be the **first synchronous action** in the click handler (before the network call is even constructed), not a side effect of the response handler. This is a common anti-pattern where developers assume the click handler's async call "feels instant" and forget that the button remains interactive until the promise resolves. This bug is closely related to BUG-004 (back-button resubmission) in that both stem from the same underlying gap: **no idempotency protection on the order-submission endpoint**, only (incomplete) client-side UI guards. Recommend:
1. Immediate fix: disable the button synchronously on click, before initiating the request, and re-enable it only on error (not on success, since success navigates away).
2. Structural fix (shared with BUG-004): add a server-side idempotency key so that even if two requests are somehow dispatched, only one order is created — this is the more robust fix and should not be skipped just because the UI fix reduces the visible symptom.

## Status

**In Progress** — client-side disable-on-click fix has been implemented and is in code review; the shared server-side idempotency key fix (tracked jointly with BUG-004) is still pending and should be verified before this bug is fully closed.
